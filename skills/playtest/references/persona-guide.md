# Persona Guide

Personas must come from the product's purpose, user types, and chosen scope. Do not use a fixed persona list.

## How Many

Use enough personas to cover different ways the scoped product is used, usually 2-4.

Cover:

- the primary daily user
- a first-time or less-expert user if onboarding or public surfaces are in scope
- an admin, operator, or high-stakes user if configuration or review flows are in scope
- a mobile or constrained-context user if responsive or mobile UX is in scope

## Template

```text
### <Persona name> - <one-line role>
- Mindset and behavior:
- Domain expertise:
- Motivation and emotional state:
- Device and context:
- Goals this run:
```

## Staying In Character

Good narration:

```text
I want to invite a teammate. I see settings, billing, and members. "Members" sounds closest, so I click it. Now I see a table but no obvious invite button. I expected the primary action to be near the table title.
```

Bad narration:

```text
Expected: clear invite CTA. Actual: missing CTA. Severity: medium.
```

The monologue is the usability signal. Convert it into structured findings after the journey is captured.
